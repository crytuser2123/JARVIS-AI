Platform Architecture & Build Specification

J.A.R.V.I.S. — a local-first voice AI assistant

A complete, build-ready brief for Claude Code to implement the platform exactly as prototyped. Cross-platform native shell, cinematic UI, and a cheap voice loop powered entirely by an OpenAI API key.

TARGET: Claude Code SHELL: Tauri 2 AI: OpenAI API PLATFORMS: macOS · Windows · Linux · Android DATE: 2026-06-08

00 How to use this document

This is the single source of truth. Build top-to-bottom; each section is a contract.

  • Implement in the order of §15 (sprints). Ship the loader + shell first — it is the product's signature.
  • The reference prototype (HTML/CSS/JS) demonstrates every screen and interaction. Treat its visual output and behavior as the acceptance target; this doc defines the production architecture behind it.
  • Anything marked MUST is non-negotiable. SHOULD is strongly preferred. MAY is optional.
  • The whole AI surface runs on one OpenAI API key held by the user. No other paid services. No telemetry.
  • When a decision is ambiguous, prefer: local-first, cheapest OpenAI model that meets quality, and the smallest reviewable change.

01 Product overview & principles

A personal, cinematic, J.A.R.V.I.S.-style desktop/mobile assistant that talks, helps with work, and lives on the user's machine.

The assistant boots with a dramatic full-screen sequence, presents a single dark glass interface with an audio-reactive particle orb and a tabbed work surface (chat, tasks, goals, research, mail, finance, tracker, knowledge, connectors, agents). It listens, thinks, calls tools, and speaks back.

Principles (MUST)

  • Local-first / private. Data lives on device. The only network egress is to the OpenAI API, on the user's key, and only when a feature needs it.
  • One key, cheap. STT + LLM + TTS + embeddings all run through the OpenAI API. No realtime API; stream a classic pipeline to feel live at a fraction of the cost.
  • Beauty is the product. Animation, sound design and "vibe" are first-class. 60fps, GPU-only transforms.
  • Modular skills. Every capability is a function-calling tool with a JSON schema.
  • Secure by default. Any system access (files, screen, mouse) is an opt-in, permission-gated module, off by default.
  • Never break the app. The loader must always reveal the UI (try/catch + hard timeout). API failures surface to the user, never swallowed.

05 Cinematic loader

The signature. A phase machine, ~7s of synthesized sound then silence, with a hard failsafe.

Phases (in order) — MUST

  1. Kernel flood (~1.5s): full-screen monospace boot-log, neon, rolling buffer (old lines dropped — never accumulate thousands of nodes). Low rumble + typing ticks.
  2. Glitch logo (~1.4s): product name large, RGB-split (clip-path + dual text-shadow) + shake keyframes. Heavy impact hit.
  3. Welcome (~1.4s): types Welcome back, <Name>. (name from Settings).
  4. Full-screen orb formation (~2.2s): ~900 particles converge from screen edges into a glowing sphere; shockwave rings; climax flash + “SYSTEM ONLINE”. Ominous rumble + dissonant riser + metallic shrieks + sub-bass boom on climax.
  5. Reveal: overlay dissolves, UI sections materialize by stagger (topbar → orb → panel).
Audio rule

All loader sound MUST fade to silence by 7 seconds (schedule a master-gain ramp on the audio clock). The climax boom lands before 7s. After boot, restore master gain so app SFX work. No background music — sound design only.

Failsafe (MUST)

Wrap the phase loop in try/catch → forceReveal(). Add a hard setTimeout(forceReveal, 16000). Respect prefers-reduced-motion with an instant path. The UI is hidden statically via a booting class in markup (no first-frame flash); reveal sets inline opacity/transform so a stalled CSS transition can never trap the app.

WebAudio synth (MUST, no files)

Provide tone() (osc + freq sweep + ADSR), noise() (filtered buffer), digi() (square + bitcrush WaveShaper), glitch(), rumble(), riser(), boom(), plus a generated ConvolverNode reverb. Unlock AudioContext on first pointerdown/keydown.

07 The Orb

Generative particle sphere, audio-reactive, with four states.

  • Render: Canvas 2D, ~460 particles on a rotating sphere projected with perspective; central radial glow + core disc; breathing.
  • States: idle · listening · thinking · speaking — each drives base hue, spread, glow and amplitude target.
  • Audio reactivity (MUST): attach an AnalyserNode to the TTS playback output; feed amplitude into the orb's scale/glow so it pulses with the assistant's voice. During listening, feed mic amplitude.
  • Expose ORB.set(state) and ORB.amp(0..1).

08 Voice pipeline (OpenAI, cheap, streamed)

Classic STT → LLM(tools) → TTS loop, streamed end-to-end to feel realtime — no Realtime API.

🎤 Mic → [wake word?] → [VAD] → [STT] → [LLM + tools] → [TTS] → 🔊 ▲ │ └──── barge-in (cut TTS on speech) ──┘
StageOpenAIImplementation notes
Wake word— (local)openWakeWord/Porcupine MAY trigger “Hey Jarvis”. Optional.
VAD— (local)Energy + WebRTC/Silero VAD. Silence window 1.2–1.5s before end-of-turn (MUST) — shorter cuts people off. Raise mic threshold to avoid false triggers.
STTgpt-4o-mini-transcribe (or whisper-1)POST captured utterance to /v1/audio/transcriptions. Stream partials where supported; otherwise transcribe per finalized turn.
LLMgpt-4o-mini (tools)/v1/chat/completions with stream:true and tools. Stream tokens to chat UI; handle tool_calls deltas.
TTSgpt-4o-mini-tts (or tts-1)/v1/audio/speech. Synthesize per sentence: speak sentence 1 while generating sentence 2.

Turn-taking & latency (MUST)

  • Barge-in: if the user starts speaking while TTS plays, immediately stop playback and start a new turn.
  • Stream everything: partial STT, LLM tokens as generated, TTS by sentence. Small models + streaming ≈ realtime feel at a fraction of the cost.
  • Cost control: cap context window, summarize old turns, and only pass the top-N relevant tools (see §09).

09 Tool / skill framework

Every capability is a function-calling tool. The core routes calls, executes, returns human-readable results, and emits a UI event.

// tool definition the model sees (keep description ≤ ~200 chars)
{
  "type": "function",
  "function": {
    "name": "task_add",
    "description": "Add a task. Call when the user asks to create/record a task.",
    "parameters": {
      "type": "object",
      "properties": {
        "title": { "type": "string" },
        "board": { "type": "string", "description": "board id, default 'main'" },
        "column": { "type": "string", "enum": ["backlog","today","progress","done"] },
        "due": { "type": "string", "description": "YYYY-MM-DD, optional" }
      },
      "required": ["title"]
    }
  }
}

Rules (MUST)

  • Tool descriptions short; long behavior lives in the system prompt.
  • When many tools exist, pass only the top-N relevant to the model (keyword/embedding match on the user turn) — keeps context lean and the model accurate.
  • Every tool result is human-readable. Never swallow errors — surface them in chat.
  • Each successful call emits a Tauri event so the relevant panel re-renders live.
  • No shell execution in the base build (see §13/§17).

11 Agents

User-built sub-agents derived from Jarvis, each with its own role, prompt, tools and knowledge.

  • List: agent cards (hexagon icon, role, prompt preview, stats: tools / knowledge bases / sources) with Run agent + edit.
  • Builder: Identity (name, color, auto glyph) · Primary role (default directive) · System prompt · Capabilities (which tools it may use) · Knowledge bases (links to vault collections / KB paths, added separately) · Sources to study (repos / GitHubs / URLs it should review).
  • Run: opens a modal chat scoped to that agent — its header, a greeting referencing its loaded KBs/sources, suggestion chips, and a task composer. Tasks run through the orchestrator with the agent's prompt + restricted tool set + its KB/source context injected.

12 Data model

SQLite for structured records; markdown vault for knowledge; localStorage for UI prefs.

CREATE TABLE boards   (id TEXT PRIMARY KEY, name TEXT);
CREATE TABLE tasks    (id TEXT PK, board_id TEXT, col TEXT, title TEXT,
                         pri TEXT, tag TEXT, due TEXT, created INTEGER);
CREATE TABLE goals    (id TEXT PK, horizon TEXT, title TEXT, sub TEXT, pct INTEGER);
CREATE TABLE accounts (id TEXT PK, name TEXT, balance REAL, color TEXT);
CREATE TABLE categories(id TEXT PK, name TEXT, budget REAL, spent REAL, color TEXT);
CREATE TABLE txns     (id TEXT PK, kind TEXT, cat TEXT, acct TEXT,
                         amount REAL, label TEXT, ts INTEGER);
CREATE TABLE habits   (id TEXT PK, day TEXT, name TEXT, done INTEGER);
CREATE TABLE agents   (id TEXT PK, json TEXT);          -- serialized record
CREATE TABLE messages (id TEXT PK, role TEXT, content TEXT, ts INTEGER);
  • Vault: each note is a .md file with YAML frontmatter (type, date, tags, [[links]]). Two-way sync: edits in-app ↔ edits in the file. Obsidian-compatible.
  • Semantic search: embed notes with text-embedding-3-small; store vectors in a local index (sqlite-vss or flat cosine for the prototype).
  • Live UI: core emit on any write → panels listen and re-render. Persist active tab/entity.

13 Security & privacy

  • Secrets in keychain (MUST): OpenAI key + OAuth tokens go to OS keychain (Keychain / Credential Manager / libsecret), never plaintext, never in the bundle.
  • Local by default; cloud opt-in. Only OpenAI egress, on the user's key.
  • Computer control (files / screenshot / mouse / app automation) is a separate, permission-gated module, off by default, with per-capability toggles, a pre-action confirmation, an audit log, a global kill-switch, and permission timeouts. No arbitrary shell in the base build.
  • Risky actions (delete, send, payments) require explicit “yes”.
  • No telemetry.

Key handling (MUST)

// key resolution order, in Rust core only
1. OS keychain  ("jarvis/openai_api_key")      // primary, set via Settings
2. env OPENAI_API_KEY                          // dev fallback
// NEVER: bundled in src/, logged, or sent to the WebView.

Streaming bridge

// frontend
await invoke("chat_send", { text, mode });   // kicks off a turn
listen("chat_token", e => appendToBubble(e.payload));   // streamed tokens
listen("tool_call",  e => renderToolCard(e.payload));
listen("tts_amp",    e => ORB.amp(e.payload));         // drives orb
Cost & resilience

Cap max tokens; summarize history beyond N turns; pass top-N tools only; retry with backoff on 429/5xx and show a clear inline error. Show a tiny live cost/usage readout (the prototype's status chips). Offer a model override in Settings.

Sprint order (ship in this sequence)

  1. Shell + loader + design system — the signature; build first. Include WebAudio SFX + failsafe.
  2. Chat + LLM (text) — orchestrator, function-calling, token streaming.
  3. Voice — STT + VAD + TTS, turn-taking (1.2–1.5s), barge-in, orb states.
  4. Skills framework + first tools — tasks/boards, goals, reminders.
  5. Knowledge vault — markdown sync + embeddings search.
  6. Panels — research, mail, finance, tracker, connectors.
  7. Agents — builder + run-chat.
  8. Computer control (permission-gated, optional) + multi-platform packaging + auto-update.

16 Definition of done

The build is accepted when every box is true.

  • Loader runs all phases, audio silent by 7s, never traps the UI (failsafe + reduced-motion verified).
  • Orb renders 4 states and pulses with live TTS amplitude.
  • Full voice loop works on an OpenAI key: speak → transcribe → stream reply → speak back, with barge-in and 1.2–1.5s turn-taking.
  • The OpenAI key is only ever in the Rust core / keychain; grep of the web bundle finds no key and no api.openai.com call.
  • Every module matches the prototype: kanban boards + DnD, goals rings, research, mail connect-gate + drafts-only, CoinKeeper finance with live balances, tracker, vault semantic search, connectors, agents builder + run-chat, full settings page.
  • At least 8 tools registered; tool calls update panels live via events; errors surface in chat.
  • Data persists across restarts (SQLite + vault); vault is Obsidian-compatible.
  • Computer-control stays off by default; no shell in base build; no telemetry.
  • Builds run on macOS, Windows, and Android from one codebase.

17 Out of scope

  • OpenAI Realtime API (we stream a classic pipeline instead).
  • Arbitrary shell / root access in the base build (only the optional permission-gated module of §13).
  • Any non-OpenAI paid AI provider. The stack stays single-key.
  • Auto-sending email or executing payments without explicit confirmation.
